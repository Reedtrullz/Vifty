Vifty's compatibility claims are evidence-based. A model family is considered validated only when a real machine report includes the read-only readiness JSON, helper fan probe output, install source, macOS version, bundle/signing evidence, and manual smoke-test result when the hardware is supported.

Use the GitHub **Hardware Validation Report** issue template to contribute evidence. The validation procedure lives in [hardware-validation.md](hardware-validation.md), and `scripts/collect-validation-evidence.sh` can gather the standard read-only evidence bundle from an installed app.

Install source matters. For `v1.1.1`, source builds from the tag and `Vifty-v1.1.1-unsigned-dev.zip` reports can contribute hardware compatibility evidence, but they are not Developer ID signed, notarized, Homebrew-trusted, or trusted public binary evidence. Current `main` or other local ad-hoc builds can contribute candidate compatibility evidence only when they are recorded as `local-ad-hoc-build` with the exact source ref and 40-character commit SHA; do not relabel post-release local builds as `source-build-tag`. Use the collector's `install-provenance.tsv` and the reviewed `installSource` / `sourceRef` / `sourceSHA` / `sourceArtifactSHA256` fields to keep that distinction visible in indexes.

## Current Claim

Vifty targets Apple Silicon MacBook Pro models on macOS 15+. The published application is arm64-only. Intel Macs cannot execute this binary, so Rosetta or a read-only safe-block claim is not Intel compatibility evidence; a future universal build would require a separate Intel build, launch, diagnostic, and unsupported-hardware validation program.

The exact public Vifty v1.3.2 build 7 now has one validated report for model identifier `MacBookPro18,1` on macOS 26.5.2. Treat every other model identifier and Vifty build as **needs validation** until its own reviewed evidence exists. Unsupported Macs should remain under macOS automatic fan control. The canonical unsupported-machine behavior is documented in [unsupported-hardware.md](unsupported-hardware.md).

## Evidence Rules

- Always capture `viftyctl diagnose --json` before any manual smoke test. It is read-only and should say whether the machine is `ready`, `degraded`, or `blocked`.
- Always capture `sudo /Applications/Vifty.app/Contents/MacOS/ViftyHelper probeLocal` for supported MacBook Pro validation, because it shows fan count, per-fan min/max RPM, `hardwareMode`, `hardwareModeRawValue`, `hardwareModeKey`, and `targetRPM` telemetry.
- Private SMC/HID drift is compatibility evidence, not a prompt to experiment with writes. If fan topology, sensors, RPM ranges, fan mode keys, mode values, target RPM telemetry, or write behavior differ from known reports, keep the machine in macOS Auto and collect read-only evidence before changing support claims.
- Do not run the manual smoke test when readiness is `blocked`, when `daemonControlPathReady` is false, when `manualControlActive` is true, when thermal pressure is critical, when fans/sensors are missing, or when RPM ranges look invalid. Follow [unsupported-hardware.md](unsupported-hardware.md) for unsupported-hardware safe-block reports.
- A `degraded` report can still be useful, but the reason must be explained. Examples include missing optional fan mode telemetry, System-managed fan mode, or an already-active lease.
- Supervised `viftyctl run` smoke evidence is useful developer-workload proof when readiness is `ready` or safely `degraded`: it should show the CLI advertised schema version `1`, the stable capabilities, diagnose, command-error, and run schema IDs, wrapper resource discovery, the safe `runLifecycle` contract, `policyStatusAvailable: true`, `policy.enabled: true`, and policy/metadata limits through `capabilities --json`, the child command was validated before cooling, diagnose readiness used schema version `1` or any nonzero diagnose command error matched the advertised command-error schema identity, one bounded lease was created, Auto was restored after the child exited, and `status --json` / `audit --limit 20 --json` / `diagnose --json` stayed machine-readable afterward. Captured summaries must prove that through `run.schemaVersion=1`, `run.schemaID=https://vifty.local/schemas/viftyctl-run.schema.json`, `run.command=run`, `run.coolingLeasePrepared=true`, `run.autoRestoreAttempted=true`, `run.autoRestoreSucceeded=true`, and `run.childExitCode=0`; when `preflight.resolvedChildExecutableReported=true`, the captured summary must also include basename-only `run.resolvedChildExecutable` with `run.resolvedChildExecutablePathPrivacy=basenameOnly`, and that display value must match the basename of the final `viftyctl-run*.json` raw absolute resolved executable while preserving current run JSON `run.childTerminationReason`, `run.childSignal`, `run.childSignalName`, and `run.resolvedChildExecutableSHA256Status` when those fields are present. Reports are accepted only when the adjacent smoke bundle has matching `manifest.tsv`, per-command `.status` files, a `privacy-review.tsv` row with status 0 and no `redaction-needed` findings, and `checksums.tsv` coverage for the summary, manifest, stdout/stderr, status files, and privacy-review files. If the run smoke follows a manual smoke test, wait for `policy.prepareCooldownSeconds`; a first `PREPARE_RATE_LIMITED` response with `retryAfterSeconds` is cooldown evidence to record and retry once, not a failed agent-run smoke test. The collector records that exact case in `rateLimitRetry` and stores the final proof in `viftyctl-run-retry.json`, with no busy-loop retries. The reviewer requires `rateLimitRetry` to match the initial `PREPARE_RATE_LIMITED` JSON and requires the final `run` proof to reference the `viftyctl-run-retry` command files. It does not replace the supported-hardware `manualSmokeTestResult: "passed-auto-restored"` requirement for validated hardware claims.
- For `v1.1.1`, record whether the install source was a source build from the tag or the unsigned-dev convenience zip. Treat those as compatibility-only evidence, not release-trust evidence. Prefer collector runs with `--install-source`, `--source-ref`, `--source-sha`, and `--source-artifact` when the unsigned zip is still available locally.
- A Homebrew or GitHub Release report should include the collector's `release-artifact-summary.json` / `release-artifact-summary.tsv` when `--release-summary` is supplied, `release-checklist.md` / `release-checklist.tsv` when `--release-checklist` is supplied, `review-summary.tsv`, `review-summary.json`, `install-provenance.tsv`, `bundle-executables.tsv`, `schema-resources.tsv`, `capabilities-schema-resources.tsv`, `capabilities-contract.tsv`, `viftyctl-audit.json`, the app/CLI/helper/daemon signing files, notarization/Gatekeeper files, bundled LaunchDaemon plist, and whether the app was installed from the expected `Vifty-v<version>.zip` artifact. The install-provenance row should be `0`, but source-first or unsigned-dev install sources are not release-trust proof. The release-summary row should be `0` only when the verifier passed, matched the installed app version, did not skip or fail individual checks, and kept SHA/artifact-name fields consistent. The release-checklist row should be `0` only when the checklist version matches the installed app and includes the required post-publication trust follow-up. The capabilities-contract row should be `0` so agent wrappers can rely on safe `runLifecycle`, direct prepare/restore lifecycle, `policyStatusAvailable: true`, wrapper resource discovery, metadata limits, and force-retry discovery fields.
- Before adding a report link to this page, run `make validation-evidence-review VALIDATION_EVIDENCE_BUNDLE=<evidence-dir> VALIDATION_EVIDENCE_REVIEW_MODE=supported-hardware VALIDATION_EVIDENCE_REVIEW_SUMMARY=<evidence-dir>/review-result.json` for supported Apple Silicon MacBook Pro claims, `VALIDATION_EVIDENCE_REVIEW_MODE=unsupported-hardware` for expected-blocked reports, or `VALIDATION_EVIDENCE_REVIEW_MODE=release` for installed public-release trust evidence. The Make target wraps `scripts/review-validation-evidence.sh`; the reviewer is read-only, checks only captured files, and writes a machine-readable `review-result.json` pass/fail summary with `schemaID: https://vifty.local/schemas/validation-review-result.schema.json`. After the issue template records **Passed and Auto restore confirmed**, rerun supported-hardware review with `VALIDATION_EVIDENCE_MANUAL_SMOKE_RESULT=passed-auto-restored VALIDATION_EVIDENCE_MANUAL_SMOKE_SOURCE=<issue-url>` so the report can be indexed as validated hardware evidence. If the supervised `viftyctl run` smoke bundle is available, include `VALIDATION_EVIDENCE_AGENT_RUN_SMOKE_SUMMARY=<smoke-dir>/agent-run-smoke-evidence-summary.json` so the reviewer validates `schemaID: https://vifty.local/schemas/agent-run-smoke-evidence-summary.schema.json`, verifies the adjacent manifest/status/privacy-review/checksum files, and indexes developer-workload proof separately from the hardware support gate. If only issue-template text is available, use `VALIDATION_EVIDENCE_AGENT_RUN_SMOKE_RESULT=passed-auto-restored VALIDATION_EVIDENCE_AGENT_RUN_SMOKE_SOURCE=<issue-url>#agent-run-smoke`; for `local-ad-hoc-build` reports also pass `VALIDATION_EVIDENCE_AGENT_RUN_SMOKE_READINESS_SUMMARY=<path>` so the reviewer records `agentRunSmokeReadinessSource` from the read-only preflight.
- Use `scripts/summarize-validation-reports.sh --input <reports-dir> --output-json <reports-dir>/compatibility-index.json --output-tsv <reports-dir>/compatibility-index.tsv --output-markdown <reports-dir>/compatibility-matrix.md` to build schema-backed report indexes and a conservative Markdown matrix draft from valid `review-result.json` files. The JSON index declares `schemaID: https://vifty.local/schemas/validation-report-index.schema.json`; the indexer rejects review results with a missing or drifted validation-review-result schema ID, missing or malformed `generatedAtUTC`, missing `daemonControlPathReady`, missing or unsupported `recommendedAgentAction`, missing or unsupported `recommendedRecoveryAction`, malformed `manualControlActive`, malformed `failedCheckIDs` / `coolingBlockerIDs`, malformed `agentRunSmokePrivacyReviewStatus`, non-empty `coolingBlockerIDs` paired with `safeToRequestCooling=true`, missing `manualSmokeReadinessSource` for passed local-ad-hoc manual smoke, missing `agentRunSmokeReadinessSource` or a captured agent-run smoke summary source for passed local-ad-hoc agent-run smoke, malformed, non-read-only, cooling-mutating, unsupported-mode, unsupported install-source, release-mode source-first install sources, invalid or missing required source SHA/checksum, mutable or missing source refs for source-build tag evidence, or contradictory passed review results. Supported-hardware rows are intentionally labeled `supported-hardware-evidence-needs-manual-smoke` until the review result includes `manualSmokeTestResult: "passed-auto-restored"`, at which point they become `validated-hardware-evidence`. The index also carries `installSource`, `sourceRef`, `sourceSHA`, `sourceArtifactSHA256`, `reviewGeneratedAtUTC`, `modelFamily`, `recommendedAgentAction`, `recommendedRecoveryAction`, `failedCheckIDs`, `coolingBlockerIDs`, `daemonControlPathReady`, `manualControlActive`, `manualSmokeReadinessSource`, `agentRunSmokeResult`, `agentRunSmokeSource`, `agentRunSmokeReadinessSource`, `agentRunSmokePrivacyReviewSource`, `agentRunSmokePrivacyReviewStatus`, `agentRunSmokeStartupMode`, `agentRunSmokeStartupModeSource`, and `agentRunSmokeStartupModeReadError` so compatibility evidence does not get confused with trusted binary evidence, helper readiness evidence, review freshness, model-family coverage, the agent cooling decision, hard cooling blockers, local manual ownership, required recovery action, manual-smoke preflight provenance, agent-run evidence, agent-run readiness provenance, captured smoke privacy-review provenance, or saved startup-mode recovery context. Legacy review results that predate `manualControlActive`, blocker-ID arrays, manual-smoke readiness provenance, agent-run readiness provenance, privacy-review provenance, or agent-run startup-mode diagnostics are indexed as blank/unknown rather than inferred. `modelFamily` is the stable prefix derived from `modelIdentifier`, such as `MacBookPro18` from `MacBookPro18,3`; use it for grouping reviewed reports, not as a chip-generation claim. The generated `compatibility-matrix.md` ignores release-trust rows for hardware status, keeps agent-run smoke evidence separate from manual-smoke validation, shows `agent-run privacy` only when a captured smoke bundle review recorded `privacy-review.tsv`, and shows `agent-run startup` only as recovery context when reviewed smoke evidence includes app preferences. Its JSON summary includes `countsByModelFamily`, `validatedHardwareReportsByModelFamily`, `countsByRecommendedAgentAction`, `countsByRecommendedRecoveryAction`, `countsBySafeToRequestCooling`, `countsByDaemonControlPathReady`, and `countsByManualControlActive` for quick safety triage across many reports.

## Before Broad Support Claims

Before calling a release broadly Apple Silicon MacBook Pro ready, collect at least:

- five validated Apple Silicon MacBook Pro reports across multiple model families;
- at least two reports from M3/M4 or newer Pro/Max machines;
- one clean Homebrew cask install report from a fresh machine;
- one report proving unsupported hardware blocks safely.

Do not add a README compatibility badge or broad marketing claim until the status table has real report links.
